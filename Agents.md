# Руководство для агентов MyMeteo

## Назначение проекта

MyMeteo - мобильное приложение для пользователей с метеозависимостью. Оно заранее предупреждает о погодных и геомагнитных изменениях, помогает вести дневник самочувствия и дает персональные рекомендации по подготовке к потенциально тяжелым дням.

Основные триггеры из постановки задачи:

- резкие скачки атмосферного давления;
- геомагнитные бури и индекс Kp;
- высокая или резко меняющаяся влажность;
- перепады температуры более чем на 7-10 градусов за сутки;
- скорость ветра и погодные фронты;
- индивидуальные реакции пользователя, выявленные по дневнику самочувствия.

Ключевая идея продукта - объединять погодные, геомагнитные и персональные факторы в единый индекс самочувствия, который объясняет пользователю, почему день может быть обычным, внимательным или рискованным.

Обычные погодные сервисы показывают температуру и прогноз, но не объясняют, как погодные изменения могут повлиять на самочувствие конкретного человека. MyMeteo должен закрывать именно этот разрыв: анализировать данные, выявлять потенциально опасные изменения, предупреждать заранее и давать спокойные практические рекомендации.

## Требования из исходных документов

Этот файл объединяет требования из всех присланных материалов. Если агент дорабатывает проект, он должен учитывать не только общую архитектуру ниже, но и конкретные пункты из этих документов.

### Проектирование функционала

- Разработать логику скоринга риска: единая формула должна объединять давление, геомагнитную бурю, влажность, температуру и персональные реакции в индекс самочувствия.
- Спроектировать хранение истории метеоданных, чтобы позже сопоставлять погоду с самочувствием пользователя.
- Сделать быстрый метео-дневник: запись состояния должна занимать не больше 5 секунд.
- Для ввода состояния использовать простую шкалу 1-10, смайлы или другой быстрый формат.
- В дневнике предусмотреть выбор симптомов: головная боль, сонливость, боли в суставах, изменение давления и дополнительные пользовательские заметки.
- Автоматически привязывать к записи дневника погодные условия на момент записи: давление, влажность, температуру, город и индекс Kp.
- Через 14-30 дней записей искать связи между конкретными факторами и самочувствием, например между влажностью и ухудшением состояния.
- Для пуш-уведомлений нужна матрица шаблонов под разные события: магнитная буря, скачок давления, мороз, резкое изменение температуры, высокая влажность.
- Уведомления делятся на информационные и срочные.
- Пользователь должен сам выбирать, о каких триггерах получать уведомления.
- Гео-триггеры и фоновое обновление местоположения используются только для актуального прогноза по текущему городу и только после согласия пользователя.
- Нужна стартовая анкета: возраст, тип метеочувствительности, хронические заболевания.
- Медицинские и чувствительные данные хранить с учетом приватности и применимого закона, например 152-ФЗ или GDPR.
- Советы в ленте должны меняться по профилю: например, гипертоникам осторожно показывать рекомендации про снижение соли, гипотоникам - про отдых, воду или чай без обещаний лечебного эффекта.

### Техническая реализация, упрощенный вариант

- Для MVP допустима упрощенная архитектура без сложного сервера.
- Можно использовать Make или Zapier для прототипа связи погодного API с базой данных, но не для финальной обработки чувствительных медицинских данных.
- Для базовых функций допускаются прямые запросы из Android-приложения к погодному API, если выбранный сервис разрешает клиентскую схему.
- Для простого прототипа правила риска могут быть табличными или условными, например: если давление ниже 740 мм рт. ст., показать высокий риск.
- Простые правила можно хранить в Firebase или Supabase.
- Базовые push-уведомления можно делать через Firebase Cloud Messaging вручную, по расписанию или полуавтоматически.
- Для интерфейса использовать готовые UI-компоненты и спокойные темы, например Material Design.
- В MVP главный экран может быть очень простым: крупный индекс риска и кнопка "Мне плохо".
- Упрощенный дневник может быть одной проверкой состояния: "Как вы?" с вариантами "Хорошо", "Средне", "Плохо".
- Для снижения когнитивной нагрузки допустима цветовая индикация всего экрана: мягкий зеленый, желтый или нежно-красный фон по уровню риска.
- Вместо нейросети на первом этапе можно использовать счетчик совпадений: сколько раз пользователь нажал "Плохо" при низком давлении, магнитной буре или другом факторе.
- В профиле можно показывать простой статистический вывод, например: "В 80% случаев вы чувствовали себя плохо при магнитных бурях".

### Техническая реализация, расширенный вариант

- Серверная часть является центром обработки данных.
- Нужны регулярные процессы сбора данных из метео-API и геомагнитных источников, например OpenWeatherMap, Open-Meteo, NOAA SWPC и других надежных сервисов.
- Сырые данные должны сохраняться в базе, чтобы risk-engine мог пересчитывать индекс благополучия.
- Risk Engine должен прогонять свежие данные через формулу индекса в режиме, близком к реальному времени.
- API для мобильного приложения должен покрывать регистрацию, профиль, дневник, персональные прогнозы, рекомендации и настройки уведомлений.
- Для пуш-уведомлений нужна очередь или планировщик: сообщения отправляются автоматически, если новые данные превысили заданные пороги.
- Интерфейс должен быть антистрессовым: приглушенные синие и зеленые тона, без резких анимаций и ярких вспышек.
- Главный экран должен иметь крупный индикатор статуса, например цветовое кольцо или индекс, который считывается одним взглядом.
- Метео-дневник должен использовать крупные кнопки, чтобы по ним было легко попасть при плохом самочувствии.
- Графики давления и температуры должны быть адаптивными, плавными и не перегруженными лишними линиями.
- Для ML-модуля нужно подготовить очищенный датасет: погода плюс записи конкретного пользователя.
- Допустимые модели персонализации: случайный лес, простая нейросеть или Java-библиотеки Smile, Tribuo, Weka.
- Модель должна выявлять веса факторов для конкретного человека.
- Обязательна проверка точности: насколько предсказанный "плохой день" совпал с реальной жалобой пользователя.
- Если модель нашла корреляцию, в интерфейсе должен появляться понятный блок, например: "Ваш персональный риск выше среднего".

### Контент и рекомендации

- Рекомендации должны опираться на авторитетные источники: ВОЗ, РЛС, клинические рекомендации Минздрава и другие проверяемые справочники.
- Источники нужно фиксировать в документации, чтобы медицинские утверждения были проверяемыми.
- Рекомендации делятся на категории: физическая активность, питание и гидратация, бытовые советы, сон, самонаблюдение.
- Примеры рекомендаций: заменить кардио на легкую растяжку, следить за водным балансом, снизить соль при подходящем профиле, проветрить комнату, поддерживать комфортную влажность, соблюдать режим сна.
- Медицинский язык нужно переводить в простой и понятный тон "заботливого помощника".
- Один совет должен быть коротким, примерно 150-200 знаков.
- Обязателен юридический дисклеймер: советы носят информационный характер и не заменяют очную консультацию врача.
- Цветовое кодирование индекса: зеленый - норма, желтый - внимание, красный - опасно.
- Желтый уровень возможен, когда отклонен один фактор, например Kp = 4, или есть умеренные колебания.
- Красный уровень возможен при сочетании триггеров, например "барометрическая пила" плюс магнитная буря.
- Графики должны показывать время по горизонтали и уровень стресса для организма по вертикали.
- Главный виджет должен быть компактным и понятным, например: "Сегодня 4/10 - умеренный риск".
- Для факторов нужны понятные иконки: солнце, волны давления, термометр, влажность, магнитная буря.
- Иконки факторов могут менять цвет в зависимости от уровня угрозы.
- Выдача советов строится через теги: фактор риска, профиль, уровень риска, противопоказания.
- Пример правила: `давление < 740` + `профиль = гипотоник` -> показать подходящий совет из категории питания или режима.
- Перед показом нескольких советов проверять, что они не противоречат друг другу.

- Предметная область проекта - влияние погодных условий на самочувствие человека.
- Метеозависимость может проявляться через головную боль, усталость, перепады давления, снижение работоспособности и общее ухудшение самочувствия.
- Ключевые погодные факторы: атмосферное давление, температура воздуха, влажность, магнитные бури, скорость ветра и погодные фронты.
- Целевые группы пользователей: люди с гипертонией и гипотонией, хроническими заболеваниями, заболеваниями суставов, пожилые люди, метеочувствительные пользователи, люди с повышенной утомляемостью и стрессом.

- Через Open-Meteo приложение получает температуру, влажность и атмосферное давление.

- Полученные данные служат основой для проектирования, разработки и тестирования приложения.

## Целевая платформа

Проект следует вести как мобильное приложение с современным интерфейсом на Java.

Рекомендуемая базовая платформа:

- Android-приложение на Java;
- Material Components for Android для современного и спокойного интерфейса;
- серверная часть на Java 21 и Spring Boot 3 для сбора данных, расчета рисков и отправки уведомлений.

Допустимы два режима реализации:

- учебная MVP-версия: Android-приложение, прямой запрос к погодному API, локальная или облачная база, простая логика риска через условные правила и базовые уведомления через FCM;
- расширенная версия: отдельная серверная часть, регулярный сбор погодных и геомагнитных данных, движок расчета риска в реальном времени, очередь пуш-уведомлений и модуль персонализации на статистике или машинном обучении.

Интерфейс должен быть минималистичным и не раздражающим: мягкие цвета, крупная типографика, понятные статусы, минимум визуального шума. Это важно, потому что пользователь может открывать приложение при головной боли, усталости или повышенной чувствительности.

Целевые группы пользователей:

- люди с гипертонией и гипотонией;
- люди с хроническими заболеваниями;
- люди с заболеваниями суставов;
- пожилые люди;
- метеочувствительные пользователи;
- люди с повышенной утомляемостью и стрессом.

## Стек

### Мобильное приложение

- язык разработки: Java;
- платформа: Android SDK;
- интерфейс: XML-разметка, Material Components, ConstraintLayout;
- архитектура: MVVM;
- асинхронные операции: не использовать Kotlin Coroutines в Java-слое, предпочтительно ExecutorService, LiveData, RxJava 3 или WorkManager в зависимости от задачи;
- сеть: Retrofit, OkHttp;
- локальное хранение: Room;
- фоновые задачи: WorkManager;
- пуш-уведомления: Firebase Cloud Messaging;
- графики: MPAndroidChart или легкий собственный компонент для индекса благополучия;
- вариант серверной части для MVP: Firebase или Supabase для хранения профиля, дневника, правил риска и простых уведомлений;
- интеграции без кода или с минимумом кода: Make или Zapier допустимы только для прототипа сбора данных, не как финальная архитектура для чувствительных данных.

### Серверная часть

- язык разработки: Java 21;
- фреймворк: Spring Boot 3;
- API: REST и OpenAPI/Swagger;
- база данных: PostgreSQL;
- кэш и очереди: Redis, если нужны быстрые пересчеты и отложенные уведомления;
- планировщик: Spring Scheduling или Quartz;
- безопасность: Spring Security, JWT-токены доступа и токены обновления;
- уведомления: Firebase Admin SDK;
- интеграции: погодный API, геомагнитный API, при необходимости справочные медицинские источники для рекомендаций.

### Данные и аналитика

- основной источник погоды: Open-Meteo API, потому что он предоставляет прогнозы в формате JSON и может использоваться без API-ключа в некоммерческих учебных сценариях;
- запасные источники погоды: OpenWeatherMap, Meteo-France или другой надежный метеорологический API;
- источник геомагнитных данных: NOAA SWPC, центр прогнозирования космической погоды, для планетарного K-индекса и геомагнитной активности;
- модель риска: сначала понятная система правил, затем расширение через машинное обучение;
- расширение для высокого уровня проекта: Java-библиотеки Smile, Tribuo или Weka для предсказания персональных триггеров по дневнику и погодной истории.

## Архитектура

### Мобильное приложение

Android-приложение делится на слои:

- `ui` - экраны, Material-компоненты, навигация, отображение статусов риска;
- `viewmodel` - состояние экранов, обработка действий пользователя;
- `domain` - сценарии приложения: расчет локальных подсказок, валидация дневника, подготовка данных для интерфейса;
- `data` - репозитории, Room DAO, клиенты Retrofit;
- `notifications` - обработка FCM, локальные напоминания, настройки предупреждений.

Основные экраны:

- главный экран с текущим индексом благополучия;
- прогноз рисков на ближайшие дни;
- дневник самочувствия с быстрым вводом за 5 секунд или меньше;
- профиль пользователя и хронические состояния;
- рекомендации;
- настройки уведомлений;
- онбординг-анкета для возраста, типа метеочувствительности и хронических состояний.

### Серверные сервисы

Серверная часть отвечает за надежную работу с внешними источниками и единые алгоритмы риска.

Основные модули:

- сбор погодных данных - регулярная загрузка давления, влажности, температуры и ветра из метео-API;
- сбор геомагнитных данных - получение индекса Kp и сведений о магнитных бурях;
- движок риска - расчет общего и персонального уровня риска из сырых данных в режиме, близком к реальному времени;
- сервис рекомендаций - подбор советов по типу пользователя и уровню риска;
- сервис дневника - хранение записей самочувствия;
- сервис профиля - анкета, группы риска, пользовательские настройки;
- сервис уведомлений - планирование, очередь и отправка пуш-уведомлений при превышении порогов;
- основной REST API - контракт для мобильного приложения.

API для мобильного приложения должен покрывать:

- регистрацию и авторизацию пользователя;
- чтение и обновление профиля;
- создание и просмотр записей дневника;
- получение текущего индекса благополучия;
- получение персонального прогноза;
- получение рекомендаций;
- чтение и обновление настроек уведомлений.

## Модель риска

Первую версию следует делать правилами, чтобы поведение было объяснимым:

- давление резко изменилось за 24 часа - риск повышается;
- температура изменилась более чем на 7-10 градусов за сутки - риск повышается;
- индекс Kp достиг опасного уровня - риск повышается;
- влажность выходит за комфортный диапазон - риск повышается;
- сильный ветер или погодный фронт усиливает риск для чувствительных пользователей;
- пользователь относится к группе риска - вес соответствующего фактора усиливается;
- дневник показывает повторяющуюся реакцию на фактор - персональный вес фактора повышается.

Итоговый индекс благополучия отображается в простой шкале:

- зеленый - обычный день;
- желтый - стоит быть внимательнее;
- красный - высокий риск, лучше подготовиться заранее.

Персонализация должна появляться только после накопления истории дневника и погоды. Ориентир из постановки - 14-30 дней записей, после чего можно искать корреляции между самочувствием и отдельными факторами: влажностью, давлением, индексом Kp, температурой или их сочетаниями.

Пример базовой формулы для первой версии:

```text
индексРиска =
  весПерепадаДавления +
  весПерепадаТемпературы +
  весВлажности +
  весВетраИФронтов +
  весИндексаKp +
  весЧувствительностиПрофиля +
  весКорреляцииСДневником
```

Каждый вес должен иметь понятное объяснение для интерфейса и тестов. Нельзя показывать пользователю только число без причины риска.

Для MVP допустим статистический минимум вместо машинного обучения: считать совпадения между отметкой "плохо" и конкретным фактором, например низким давлением или магнитной бурей. В профиле можно показывать простую заметку вроде: "В 80% случаев вы отмечали плохое самочувствие при магнитных бурях".

Модуль машинного обучения для высокого уровня проекта строится только после появления очищенного датасета `погода + дневник пользователя`. Допустимые модели: случайный лес, простая нейросеть или Java-библиотеки Smile, Tribuo, Weka. Обязательна валидация: насколько предсказанный плохой день совпал с реальной жалобой пользователя.

Если машинное обучение или статистика выявили персональный фактор, прогноз должен показывать это отдельно: например, "Ваш персональный риск выше среднего из-за прошлых реакций на влажность".

## Данные

Ключевые сущности:

- пользователь - учетная запись;
- профиль пользователя - хронические состояния, чувствительность, настройки;
- погодный снимок - температура, влажность, атмосферное давление, ветер и другие погодные показатели по времени и локации;
- геомагнитный снимок - индекс Kp и сведения о магнитных бурях;
- запись дневника - самочувствие, симптомы, заметки пользователя;
- прогноз риска - рассчитанный риск на дату и локацию;
- рекомендация - совет, привязанный к фактору риска и профилю;
- настройки уведомлений - правила отправки уведомлений;
- шаблон уведомления - текст уведомления для конкретного события и уровня критичности;
- снимок локации - город или координаты, использованные для актуального прогноза;
- правило риска - пороговое правило или табличная логика для MVP;
- источник рекомендации - источник, на котором основан совет;
- тег рекомендации - связь совета с фактором, профилем и уровнем риска.

Для дневника обязательна автоматическая привязка метеоусловий к моменту записи пользователя: давление, влажность, температура, индекс Kp и город. Пользователь не должен вводить эти параметры вручную.

Для MVP погодный репозиторий может напрямую получать из Open-Meteo температуру, влажность и атмосферное давление, а геомагнитный репозиторий - K-индекс из NOAA SWPC. Эти данные должны сохраняться вместе с временем и городом, чтобы позднее сравнивать их с дневником самочувствия.

Базовые симптомы для выбора в дневнике:

- головная боль;
- сонливость;
- боли в суставах;
- изменение артериального давления;
- усталость;
- раздражительность или тревожность;
- свободная заметка.

## Уведомления

Пуш-уведомления делятся на два уровня:

- информационные - прогноз на завтра, умеренный риск, рекомендация подготовиться;
- срочные - резкое изменение давления, сильная геомагнитная буря или другой высокий риск прямо сейчас.

Для каждого типа события нужна матрица шаблонов:

- геомагнитная буря;
- скачок давления;
- резкое похолодание или потепление;
- высокая влажность;
- персональный триггер, найденный по дневнику.

Экран настроек уведомлений должен позволять включать и отключать отдельные триггеры. Гео-триггеры и фоновое обновление местоположения разрешены только после явного согласия пользователя и должны использоваться для актуализации прогноза в текущем городе.

## Контент и рекомендации

Рекомендации должны опираться на проверяемую базу источников. В документации фиксировать ссылки или названия источников, например:

- рекомендации ВОЗ;
- РЛС;
- клинические рекомендации Минздрава;
- другие авторитетные медицинские справочники.

Категории советов:

- физическая активность: снижение нагрузки, замена кардио на легкую растяжку;
- питание и гидратация: соль, вода, регулярное питание;
- бытовые советы: проветривание, влажность в помещении, режим сна;
- самонаблюдение: дневник, измерение давления, отслеживание повторяющихся симптомов.

Правила текста:

- тон общения - заботливый помощник, без паники и давления;
- один совет - 150-200 знаков;
- язык простой, без медицинского жаргона;
- совет не должен обещать лечение или гарантированный эффект;
- рядом с медицинскими рекомендациями должен быть дисклеймер, что информация не заменяет очную консультацию врача.

Механика выдачи рекомендаций строится через теги: фактор риска, уровень риска, тип профиля и противопоказания. Пример: `давление < 740` + `профиль = гипотоник` -> совет из категории питания или режима. Перед показом нескольких советов проверять, что они не противоречат друг другу.

## Правила интерфейса

- Использовать спокойную палитру, без агрессивных красных экранов.
- Антистрессовый интерфейс: приглушенные синие и зеленые тона, без резких анимаций, ярких вспышек и перегруженных экранов.
- Крупные шрифты и хорошая контрастность обязательны.
- В MVP главный экран может состоять из большого индекса риска и кнопки "Мне плохо".
- Главный экран должен отвечать на вопрос: "Что со мной может быть сегодня и завтра?"
- Запись в дневник должна занимать не более 5 секунд: шкала 1-10, смайлы или другой быстрый выбор.
- Для упрощенного дневника допустим чек-ин "Как вы?" с вариантами "Хорошо", "Средне", "Плохо".
- Элементы дневника должны быть крупными кнопками, по которым легко попасть при плохом самочувствии.
- Рекомендации должны быть короткими, практичными, персональными и не заменять врача.
- Медицинские советы формулировать осторожно: лекарства - только по назначению врача.
- Любой высокий риск должен сопровождаться объяснением причины: давление, Kp, температура, влажность или персональный триггер.
- Динамический контент зависит от профиля: например, гипертоникам можно показать осторожный совет снизить соль, а гипотоникам - мягкую рекомендацию про отдых, воду или чай без обещаний медицинского эффекта.
- Главный индикатор должен считываться одним взглядом: крупное число, цветовое кольцо или мягкая заливка фона текущим риском.
- Цветовая шкала: зеленый - норма, желтый - внимание, красный - опасно из-за сочетания триггеров. Красный использовать мягко, без тревожной визуальной агрессии.
- Графики давления, температуры и "стресса для организма" должны быть плавными, адаптивными и без лишних линий.
- Виджет главного экрана должен показывать компактный статус, например "Сегодня 4/10 - умеренный риск", чтобы пользователь не разбирался в сырых цифрах.
- Для факторов риска использовать понятные иконки: солнце, волны давления, термометр, влажность, магнитная буря. Иконки могут менять цвет по уровню угрозы.

## Правила разработки для агентов

- Сохранять Java как основной язык реализации.
- Не смешивать интерфейс, сетевые запросы и бизнес-логику в одном классе.
- Любой новый экран строить через MVVM.
- Внешние API оборачивать в репозитории.
- Алгоритмы риска держать в отдельном доменном модуле, покрывать тестами.
- Не добавлять медицинские утверждения без источника в документации.
- Не хранить токены API в коде; использовать `local.properties`, переменные окружения или секреты CI.
- Клиентский API-ключ для прямых запросов в MVP допустим только если сервис разрешает такую схему и ключ не дает доступа к чувствительным данным.
- Для пользовательских симптомов, профиля и геолокации учитывать приватность данных.
- Медицинские и чувствительные данные хранить безопасно с учетом применимого закона, например 152-ФЗ в РФ или GDPR.
- Запрашивать разрешение на геолокацию только тогда, когда пользователь понимает, зачем оно нужно.
- Интеграции без кода или с минимумом кода не должны получать медицинские данные пользователя без отдельного обоснования и согласия.
- Любой новый совет должен иметь категорию, источник, теги выдачи и текст дисклеймера, если совет касается здоровья.

## Тестирование

Минимальный набор проверок:

- модульные тесты для движка риска;
- тесты для формулы скоринга и объяснений факторов риска;
- тесты для маппинга ответа API в доменные модели;
- тесты Room DAO для дневника;
- тесты ViewModel для главного экрана, дневника и настроек уведомлений;
- тесты для логики уровней пуш-уведомлений и пользовательских предпочтений;
- тесты для релевантности рекомендаций и отсутствия противоречий между советами;
- валидационные тесты для статистики или предсказаний машинного обучения на исторических записях;
- интеграционные тесты серверной части для REST API и планировщика сбора данных;
- ручная проверка интерфейса на маленьком экране, при крупном системном размере шрифта и в сценарии плохого самочувствия пользователя.
