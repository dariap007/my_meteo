# MyMeteo Agents Guide

## Назначение проекта

MyMeteo - приложение для пользователей с метеозависимостью. Оно заранее предупреждает о погодных и геомагнитных изменениях, помогает вести дневник самочувствия и дает персональные рекомендации по подготовке к потенциально тяжелым дням.

Основные триггеры из постановки задачи:

- резкие скачки атмосферного давления;
- геомагнитные бури и индекс Kp;
- высокая или резко меняющаяся влажность;
- перепады температуры более чем на 7-10 градусов за сутки;
- индивидуальные реакции пользователя, выявленные по дневнику самочувствия.

## Целевая платформа

Проект следует вести как мобильное приложение с современным интерфейсом на Java.

Рекомендуемая базовая платформа:

- Android-приложение на Java;
- Material Components for Android для современного, спокойного интерфейса;
- backend на Java 21 и Spring Boot 3 для сбора данных, расчета рисков и отправки уведомлений.

Интерфейс должен быть минималистичным и не раздражающим: мягкие цвета, крупная типографика, понятные статусы, минимум визуального шума. Это важно, потому что пользователь может открывать приложение при головной боли, усталости или повышенной чувствительности.

## Стек

### Mobile Frontend

- Language: Java.
- Platform: Android SDK.
- UI: XML layouts, Material Components, ConstraintLayout.
- Architecture: MVVM.
- Async: Kotlin Coroutines не использовать в Java-слое; предпочтительно ExecutorService, LiveData, RxJava 3 или WorkManager в зависимости от задачи.
- Network: Retrofit, OkHttp.
- Local storage: Room.
- Background tasks: WorkManager.
- Push notifications: Firebase Cloud Messaging.
- Charts: MPAndroidChart или легкий кастомный компонент для индекса благополучия.

### Backend

- Language: Java 21.
- Framework: Spring Boot 3.
- API: REST + OpenAPI/Swagger.
- Database: PostgreSQL.
- Cache and queues: Redis, если нужны быстрые пересчеты и отложенные уведомления.
- Scheduler: Spring Scheduling или Quartz.
- Security: Spring Security, JWT access tokens, refresh tokens.
- Notifications: Firebase Admin SDK.
- Integrations: weather API, geomagnetic API, optional medical reference source for recommendations.

### Data and Analytics

- Weather sources: OpenWeatherMap, Meteo-France or another reliable meteorological API.
- Space weather sources: NOAA SWPC or other Kp/geostorm monitoring provider.
- Risk model: deterministic scoring first, ML extension later.
- ML extension for excellent grade: Java library Smile, Tribuo or Weka for personal trigger prediction from diary and weather history.

## Архитектура

### Mobile Application

Android-приложение делится на слои:

- `ui` - экраны, Material-компоненты, навигация, отображение статусов риска.
- `viewmodel` - состояние экранов, обработка действий пользователя.
- `domain` - сценарии приложения: расчет локальных подсказок, валидация дневника, подготовка данных для UI.
- `data` - репозитории, Room DAO, Retrofit API clients.
- `notifications` - обработка FCM, локальные напоминания, настройки предупреждений.

Основные экраны:

- главный экран с текущим индексом благополучия;
- прогноз рисков на ближайшие дни;
- дневник самочувствия;
- профиль пользователя и хронические состояния;
- рекомендации;
- настройки уведомлений.

### Backend Services

Backend отвечает за надежную работу с внешними источниками и единые алгоритмы риска.

Основные модули:

- `weather-ingestion` - регулярный сбор погоды, давления, влажности, температуры.
- `space-weather-ingestion` - сбор геомагнитных показателей и Kp index.
- `risk-engine` - расчет общего и персонального уровня риска.
- `recommendation-service` - подбор советов по типу пользователя и уровню риска.
- `diary-service` - хранение записей самочувствия.
- `profile-service` - анкета, группы риска, пользовательские настройки.
- `notification-service` - планирование и отправка push-уведомлений.
- `api-gateway` или основной REST API - контракт для мобильного приложения.

## Модель риска

Первую версию следует делать правилами, чтобы поведение было объяснимым:

- давление резко изменилось за 24 часа - повышает риск;
- температура изменилась более чем на 7-10 градусов за сутки - повышает риск;
- Kp index достиг опасного уровня - повышает риск;
- влажность выходит за комфортный диапазон - повышает риск;
- пользователь относится к группе риска - усиливает вес соответствующего фактора;
- дневник показывает повторяющуюся реакцию на фактор - повышает персональный вес.

Итоговый индекс благополучия отображается в простой шкале:

- green - обычный день;
- yellow - стоит быть внимательнее;
- red - высокий риск, лучше подготовиться заранее.

## Данные

Ключевые сущности:

- `User` - учетная запись.
- `UserProfile` - хронические состояния, чувствительность, настройки.
- `WeatherSnapshot` - погодные показатели по времени и локации.
- `GeomagneticSnapshot` - Kp index и сведения о магнитных бурях.
- `DiaryEntry` - самочувствие, симптомы, заметки пользователя.
- `RiskForecast` - рассчитанный риск на дату и локацию.
- `Recommendation` - совет, привязанный к фактору риска и профилю.
- `NotificationPreference` - правила уведомлений.

## UX Guidelines

- Использовать спокойную палитру, без агрессивных красных экранов.
- Крупные шрифты и хорошая контрастность обязательны.
- Главный экран должен отвечать на вопрос: "Что со мной может быть сегодня и завтра?"
- Рекомендации должны быть короткими, практичными и не заменять врача.
- Медицинские советы формулировать осторожно: лекарства - только по назначению врача.
- Любой высокий риск должен сопровождаться объяснением причины: давление, Kp, температура, влажность или персональный триггер.

## Development Rules for Agents

- Сохранять Java как основной язык реализации.
- Не смешивать UI, сетевые запросы и бизнес-логику в одном классе.
- Любой новый экран строить через MVVM.
- Внешние API оборачивать в репозитории.
- Алгоритмы риска держать в отдельном доменном модуле, покрывать тестами.
- Не добавлять медицинские утверждения без источника в документации.
- Не хранить токены API в коде; использовать `local.properties`, переменные окружения или секреты CI.
- Для пользовательских симптомов и профиля учитывать приватность данных.

## Testing

Минимальный набор проверок:

- unit tests для risk-engine;
- tests для маппинга API response в доменные модели;
- Room DAO tests для дневника;
- ViewModel tests для главного экрана, дневника и настроек уведомлений;
- backend integration tests для REST API и планировщика сбора данных;
- ручная проверка UX на маленьком экране и при крупном системном размере шрифта.
